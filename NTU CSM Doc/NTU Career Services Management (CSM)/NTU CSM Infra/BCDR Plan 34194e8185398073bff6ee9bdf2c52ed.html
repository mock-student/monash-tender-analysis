Section 11: Key Risks and Limitations

Risk 1 — Default VPC with public subnets only: QA and Dev environments use the default VPC, which only has public subnets. ECS Fargate tasks are assigned public IPs and RDS operates in a public subnet (though publicly accessible is set to false). In a real AZ outage, recovery depends on the default VPC having subnets in multiple AZs — which AWS default VPCs do.

Risk 2 — Single ECS task (QA): The QA ECS service has desiredCount: 1, meaning complete outage during task replacement. Production is configured with desiredCount: 2 and auto-scaling (2–4), which provides better resilience.

Risk 3 — RDS DNS TTL during failover: The application's database connection (NestJS + Drizzle ORM) needs to gracefully handle transient connection failures during RDS failover. The RDS DNS endpoint doesn't change during failover, but the underlying IP does — DNS TTL is typically 5 seconds for RDS, which determines reconnection speed.

Risk 4 — RDS restore to same identifier: Database restoration from a snapshot to the same instance identifier requires deleting the existing instance first, introducing additional downtime. Alternative: restore to a new identifier and update the connection string.

Risk 5 — S3 and Cognito excluded: This plan does not test S3 or Cognito resilience, as those are regionally-managed AWS services with built-in multi-AZ durability and do not require DR drills for ISO compliance at the infrastructure layer.

This DR Drill plan covers 2 core scenarios (RDS failover + ECS task recovery) plus a bonus CI/CD validation, and can be completed within approximately 30–45 minutes on the QA environment.
