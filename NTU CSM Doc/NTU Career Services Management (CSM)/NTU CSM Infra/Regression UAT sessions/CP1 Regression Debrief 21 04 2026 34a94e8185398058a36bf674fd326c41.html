CSM-240 / Unlink Sessions Within a Bundle

Status: FIXED on dev for the main complaint. One follow-up gap — see below.

What was reported

  1. After unlinking, admin sees only one session with no way to trace the previously-linked pair.
  1. Admin needs to be able to publish a session from the session detail view.

Root cause (code-level)

  1. Old UnlinkBundleModal was a generic “Are you sure?” confirm — no session context before or after the action.
  1. SGCCSessionDetails had no publish affordance; publishing was only reachable from the sessions list.

Fix

Ticket (Linear)PRCommitRepo
SU-4043#1235ebe9a36d3platform-frontend

New flow (2026-04-21):
- UnlinkBundleModal now renders both sessions as SessionSummaryCards with a GoToIcon link to each one.
- Post-unlink success state keeps both cards visible so admin can act on either session directly from the modal.
- New SessionActionsMenu component is used in both the session detail view and the unlink modal. It exposes Publish Now and Publish Later — which fixes complaint (2) above.
- AddEditSessionModal now opens the unlink modal when a user toggles “bundled” off, using unlinkSessionSnapshot to keep the paired reference alive during confirmation.

Gaps

Persistent prior-link trace is not implemented. The fix only shows both sessions inside the unlink modal itself. Once the modal closes, session.pairedSession is null and SGCCSessionDetails has no “previously linked to X” indicator. If joji wants this visible indefinitely after unlink, we’d need a DB-level formerlyPairedSessionId (or similar) field plus a read surface in the detail view.

CSM-232 / Student Books Individual Session After Admin Unlinking

Status: FIXED on dev for copy and cancellation logic. Session-order gap — see below.

What NTU reported

2026-04-07:
- Order of session should always be resume critique first.
- Remove first line in the Important Reminders box: “Mark your attendance using the QR code provided by your coach at each session.”
- Change first line to “Students must attend both resume critique and mock interview.”
- If cancellation is toggled off, the third bullet must say “Cancellations are not allowed.” Same for Rescheduling (2nd bullet).

2026-04-20 update:
- Remove the bullet point about rescheduling.
- First bullet point change to: “Students MUST attend both Resume Critique and Mock Interview.”
- Third bullet point change to: “Cancellations are NOT allowed.”

Root cause (code-level)

  • SGCCCollapsibleReminder.tsx built the list with fallback strings like “Rescheduling is not allowed.” instead of hiding the bullet; wording didn’t match joji’s revised copy.
  • The cancel button was rendered even when cancellation was disabled, giving unintended click affordance.
  • Student-side session list is ordered by sessionDate only (sessions.service.ts:242); no secondary sort by session type.

Fix

SU-4156#1234242caaf51platform-frontend

Reminder copy rewrite (commit f0101bbcd, 2026-04-20)apps/ntu/src/containers/SGCC/StudentView/SGCCCollapsibleReminder.tsx — now matches Joji’s Apr 20 list exactly:
- Bullet 1: Students MUST attend both Resume Critique and Mock Interview. (MUST bolded)
- Bullet 2: rendered only when rescheduleEnabled — no more “Rescheduling is not allowed” fallback.
- Bullet 3: Cancellations are NOT allowed. (NOT bolded) when cancellation disabled.
- QR code line is already absent from both dev and main.

Cancel button logic (PR SU-4156)SGCCSessionCard and SGCCPage — button is disabled (not hidden) when cancellation is unavailable; click handler is gated behind the same flag.

Gaps

sessions

If two sessions share a date, the order between Resume Critique and Mock Interview is non-deterministic. Fix is a one-line change to add a secondary sort by sessionType favouring resume_critique.

Addressing questions:

1.- What happened on each failed ticket?

  • No information or coordination gap on the dev side. PRs cite the failed-test IDs directly in branch names (e.g. vitaliipetryk/su-4043-failed-test-p2-cp1-uat-se-12-unlink-sessions-within-a-bundle), meaning devs had the context needed to ship targeted fixes on the first try.
  • Gap was between QA timing and release cadence. Fixes were merged to dev on Apr 20–21; the regression build was cut against a version that pre-dated those merges.
  • Two items slipped through the fix batch (session order RC-first, persistent prior-link trace).

2.- Prevention for CP2 (raw ideas):

1. Run regression against dev (or a release-candidate branch), not against qa after the last promotion.  
Three tickets were already closed in code by the time the regression report was compiled. This single change would have converted them from “failed” to “passed.”

2. Shorten the dev → qa → main promotion cadence during UAT windows.
Current window showed ~4 days between promotions with SGCC fixes queuing behind it.

3. Enforce a structured failure description on Jira tickets.
The three original Jira exports had Expected behavior/results and Issue description/steps leading to the issue as empty fields. All the detail lived in comments, which two of the three CSV exports stripped entirely. Either a required-field template or a QA checklist covers this.

4. Keep the new habit of citing failed-test IDs in branches and PRs.
This made it trivial to map CSM-XXX → SU-XXXX → commit in this debrief. Worth making it the default.

Proposal for Action Items

  • Make a more structured pipeline on deciding what goes into qa / regression and compare side by side with what’s been promoted from dev → main.
