Section 4: Scenario B — ECS Service Recovery

Objective: Validate that ECS Fargate automatically replaces a stopped task and the ALB resumes routing traffic to healthy targets within the expected downtime budget.

Expected Downtime: 2–5 minutes (new task provisioning + image pull + 2 consecutive successful ALB health checks at default interval).

Section 9: Key Risks and Limitations

Risk 1 — Single task (desiredCount: 1). There is no redundancy; the drill guarantees a brief total outage between T1 and T4. For production workloads, set desiredCount ≥ 2 spread across both AZs and enable the deployment circuit breaker with rollback.

Risk 2 — Public subnets + public IPs for image pull. The service has no NAT gateway, so Fargate tasks must be assigned public IPs to pull public.ecr.aws/docker/library/nginx:alpine. If assignPublicIp is ever flipped to DISABLED, new tasks will fail to start with a CannotPullContainer error and recovery will stall indefinitely. Rollback step RB-2 covers this.

Risk 3 — No container logging. The task definition has no awslogs log driver configured. Failure diagnostics during the drill are limited to ECS event history and ALB metrics. Recommend adding awslogs → /ecs/ok-api before the next drill.

Risk 4 — No circuit breaker. The service was created without deployment circuit breaker + rollback. A bad task definition revision could keep failing tasks indefinitely. Recommend enabling it.

Risk 5 — HTTP only, no TLS. The ALB listener is HTTP:80 with no ACM cert. Acceptable for a demo, not for anything real.

Risk 6 — AZ of replacement task is not pinned. ECS may place the replacement task in the same AZ as the one that "failed". This drill simulates task-level failure, not AZ-level failure. A true AZ-outage test would require pulling an AZ's subnet from the service network config, which is out of scope here.

Out of scope: RDS failover (no database), CI/CD pipeline validation (no pipeline), multi-region failover (single-region by design), WAF/Route 53/ACM (not provisioned).

This DR drill plan covers one scenario (ECS task recovery) and can be completed in approximately 15–20 minutes on the ok-api demo environment.
