Section 9: Key Risks and Limitations

Risk 1 — Single task (desiredCount: 1). There is no redundancy; the drill guarantees a brief total outage between T1 and T4 (observed: 26–29 seconds of 5xx this run). For production workloads, set desiredCount ≥ 2 spread across both AZs and enable the deployment circuit breaker with rollback.

Risk 2 — Public subnets + public IPs for image pull. The service has no NAT gateway, so Fargate tasks must be assigned public IPs to pull public.ecr.aws/docker/library/nginx:alpine. If assignPublicIp is ever flipped to DISABLED, new tasks will fail to start with a CannotPullContainer error and recovery will stall indefinitely. Rollback step RB-2 covers this.

Risk 3 — No container logging. The task definition has no awslogs log driver configured. Failure diagnostics during the drill are limited to ECS event history and ALB metrics. Recommend adding awslogs → /ecs/ok-api before the next drill.

Risk 4 — No circuit breaker. The service was created without deployment circuit breaker + rollback. A bad task definition revision could keep failing tasks indefinitely. Recommend enabling it.

Risk 5 — HTTP only, no TLS. The ALB listener is HTTP:80 with no ACM cert. Acceptable for a demo, not for anything real.

Risk 6 — AZ of replacement task is not pinned. ECS may place the replacement task in the same AZ as the one that "failed". This drill simulates task-level failure, not AZ-level failure. A true AZ-outage test would require pulling an AZ's subnet from the service network config, which is out of scope here. (This drill happened to land the replacement in a different AZ — ap-southeast-1b → ap-southeast-1a — but that is not guaranteed.)

Out of scope: RDS failover (no database), CI/CD pipeline validation (no pipeline), multi-region failover (single-region by design), WAF/Route 53/ACM (not provisioned).

This DR drill plan covers one scenario (ECS task recovery) and was completed in approximately 8 minutes of active drill time on the ok-api demo environment on 2026-04-17.

Appendix A: Shell Command Notes (zsh on macOS)

A few commands in earlier drafts failed to paste cleanly into zsh because of Notion/Slack auto-formatting. Fixes applied throughout this runbook:

  • URLs must not be wrapped in angle brackets. <http://...> is treated by zsh as redirection (< = read-from, > = write-to) and produces zsh: no such file or directory. Use bare URLs: http://....
  • Pipe to head must not be escaped. \| is a literal | argument, not a pipe. Use | head -n 1.
  • JMESPath literal strings use backticks, not escaped backslashes. [?name==+""+privateIPv4Address+""+] inside single-quoted -query. Do not write \\privateIPv4Address\ or \privateIPv4Address``.
  • Do not prefix a one-liner with a label or comment. runningCount == desiredCount: while true; do ... is parsed by zsh as a command, not a heading. Remove any such prefix before pasting.
  • Re-export TASK_ARN after any auth refresh or new terminal session. The variable does not persist across aws sso login or new shells.
