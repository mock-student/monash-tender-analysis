Workspace Settings

The Workspace Settings module provides admins with tools to manage the organisational structure of the CSM platform. It contains three tabs: Work Groups, Venues, and Labels.

Work Groups

Work Groups define how users are organised within the platform and control access to modules and records.

Workgroups operate at two levels:

1. Module-level access

Admins control which workgroups can access which modules. If a user does not belong to the relevant workgroup, the module is not visible to them. For example, the SGCC module is only visible to users in the SGCC workgroup; the Mentorships module would only be visible to users in the Mentorship workgroup.

2. Record-level visibility within shared modules

For platform-wide modules (Forms, Communications, Events), users can only see records — templates, campaigns, forms — that have been explicitly shared with their workgroup.

Within any accessible module, users see all records (e.g. all Sessions, all Students) subject to role-level restrictions. Workgroup membership does not further filter records down to only those belonging to coaches within the same workgroup.

The Work Groups tab displays a table of all work groups with the following columns: Name, Admin, No. of Members. Clicking a row opens a drawer showing the work group name, its members, and each member's role in the CSM.

Filters: Keyword search (by name, creator, or member), Creator, Search by Member.

Venues

The Venues tab displays a table of all venues with the following columns: Venue, Capacity. Venue metadata includes Name, Building, Level, Room ID, Category, and Type. Capacity is an admin-editable free-text field; if empty, it displays as —.

Filters: Keyword search, Building Code, Level, Category, Type, Capacity Range.

Labels

Linear: SU-4398

Labels are a second, lighter-weight classification layer for segmenting users — distinct from Work Groups. Labels are surfaced as a dedicated tab in Workspace Settings and are automatically applied to students, staff, and admins based on declarative attribute rules (e.g. scholarship tag = YES, programme code = NHC/REP), rather than manual assignment.

The Labels tab displays a table of all labels with Name, Description, and Matched Users count. Clicking a row opens a detail view showing active attribute rules and a paginated list of matched users.

Filters: Keyword search by label name.

Attribute Rules

Each label has one or more attribute rule conditions (field + operator + value) joined by AND/OR logic. Supported operators: equals, not equals, contains, is set, is not set. Label membership is evaluated automatically — any user whose profile satisfies the rules is included. Membership updates near real-time or on next login when a user's attributes change.

Downstream usage: Labels are available as targeting primitives across Comms (M3), SGCC session filters, and Forms routing.

Dependency: The user-profile attribute extension (programme code, college, scholarship tag, year, internship status) must be schema-agreed before build begins. Data availability to be confirmed with Lily / NTU CITS.

Student Remarks

Student Remarks is a global, module-agnostic notes layer on each student's profile. Unlike SGCC session remarks (which are scoped to a specific session or SGCC registration), overall remarks are visible wherever the student's profile is accessed across the platform.

Where it lives: Users > Students tab > student detail view.

Who can add remarks: Admin, Manager, and Staff roles.

Behaviour:

  • Remarks are freeform text, timestamped, and attributed to the author.
  • All remarks are visible to Admin, Manager, and Staff — there is no per-remark visibility control.
  • Remarks are read-only after submission and cannot be edited by any role.
  • Remarks cannot be deleted by any role (retained for audit integrity).
Scope note: This section covers overall remarks only. SGCC-specific session remarks — which can be linked or unlinked to a session — are documented in 📋 SGCC § 3.9 and § 5.2.

Concurrent Edit Locking

Linear: SU-4396

When two admins or managers open the same record in edit mode simultaneously, the last save silently overwrites the other's changes. To prevent data loss, a pessimistic edit lock is implemented across all editable record types: Sessions, Events, Forms, and Campaigns.

How it works

When Admin A opens a record for editing, a lock is acquired and held until they save, cancel, or the lock TTL expires. If Admin B attempts to open the same record in edit mode, they see a non-dismissable blocking message:

"[Name] is currently editing this [session/event/form/campaign]. You can view it in read-only mode."
  • Lock TTL: 15 minutes of inactivity. A countdown warning appears 2 minutes before expiry.
  • Lock release: Immediate on save or explicit cancel. Best-effort release on tab close via beforeunload.
  • Lock holder experience: A heartbeat from the client extends the TTL while the edit form is mounted and the window has focus.
  • Blocked user experience: Placed in read-only mode (genuine render — not just disabled inputs). Notified when the lock is released so they can enter edit mode without a full page reload.
  • Lock service failure: Fails open — edit is permitted and the failure is logged.

Scope

Applies across all modules: SGCC (M1), Forms (M2), Communications (M3), and future modules.

Dynamic Column Visibility

Linear: SU-4505

Every table across the platform ships with a column picker letting users show, hide, and reorder columns to suit their workflow.

How it works

  • Each table has a default set of visible columns (for existing tables, this is the table's current column set) plus a set of optional columns, hidden by default.
  • Optional columns are added into view via an "Add Columns" dropdown. Once added, a column cannot be removed from that dropdown — the checkbox becomes permanently checked and disabled there.
  • The only way to remove an added column is via "Hide column" on that column's own header dropdown, which also contains Sort ascending, Sort descending, Reset sorting (existing functionality, relocated here), Insert field left, and Insert field right.
  • Columns can be reordered by dragging the header, or via "Insert field left" / "Insert field right".
  • Column visibility, order, and sort state are stored client-side only, scoped per user per table — preferences do not sync across devices or browsers.
  • Table content scrolls at the page level rather than within an internal container.

Scope

Applies to all existing tables at their current column sets. Any table built after this feature ships must have its default-visible and optional-additional columns explicitly defined before the column picker is implemented for it.
