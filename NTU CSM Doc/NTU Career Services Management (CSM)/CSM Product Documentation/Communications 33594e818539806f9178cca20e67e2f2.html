1. Overview

The Communications module covers all outbound emailing from the CSM platform: automated system emails triggered by platform events, admin-built campaigns delivered to selected audiences, and the reporting that follows. Audience-management infrastructure (Recipient Management and Segmentation) sits alongside delivery so admins can compose and target campaigns in one place. All current Communications features are Admin- and Manager-facing; Staff (L2) has no access to any Communications surface unless explicitly noted.

2. Admin / Manager View

2.1 System Emails

System emails are automated emails triggered by platform events. Scope is currently limited to the Users module only: new user added, staff added, staff removed.

Each system email has a trigger displayed in the admin UI showing:

  • Timing — when the email fires (e.g. "Before session", "On user creation")
  • Receiver — who receives the email (e.g. "Student", "Staff", "Admin")

The trigger display is rendered as a human-readable label translated from the backend trigger value. Active/inactive status is shown as a badge on each system email in the list. Linear: SU-4414

2.2 Campaign Delivery

For CP3, admins build and deliver email campaigns from existing templates. The campaign lifecycle covers template selection, recipient configuration, scheduling, recurrence, send confirmation, and history.

Capabilities (CP3):

CapabilityAdmin L4Manager L3Staff L2
Create a new campaign from a template✓✓ (immediate sends only)✗
Schedule a campaign for a future date/time✓✗✗
Configure recurrence rules (daily / weekly / monthly / custom)✓✗✗
Edit, pause, resume, or cancel scheduled and recurring campaigns✓✗✗
View campaign history and per-campaign delivery report✓✓✗

Recipient configuration — three methods, combinable:

  • Segment — predefined audience built from filter rules over user attributes (programme, school, year of study, graduation year, role, registration date, etc.); saved and reusable; managed in a separate Segments area. For recurring campaigns, segment membership is re-evaluated at each send.
  • List — saved curated collection of specific users; managed in a separate Lists area; created via CSV upload or by selecting users in CSM.
  • Ad-hoc selection — individual users searched and picked at campaign creation time; not saved.

Scheduling — Send now (immediate on confirmation), Schedule for later (SGT date/time, queued for delivery), or Recurring per a rule. Scheduled campaigns sit in a Scheduled list and remain editable/cancellable until send time. All times display in SGT (UTC+8).

Recurrence rules — frequency (daily / weekly / monthly / custom every N), day-of-week (weekly), day-of-month (monthly), start and end date (or no end date), time of day per send. For segment-based recurring campaigns, audience refreshes automatically at each send. Admin can pause, resume, or edit (template / recipients / future schedule) without affecting past sends.

Confirmation — before any send or schedule, the admin sees a summary (template, recipient method and count, send time / recurrence rule, attachment count, embedded polls/forms count) followed by an explicit "Send to X recipients? This cannot be undone." confirmation.

Campaign history — all past campaigns listed with name, template, send time, recipient count, delivery status, and click-through to a per-campaign performance dashboard (see 3.3). Scheduled campaigns appear separately. Recurring campaigns appear as a parent entry with each individual send beneath.

2.3 Recipient Management

For CP3, admins build and manage recipient lists for email campaigns. The lifecycle covers Excel/CSV import (with column mapping and validation), in-place review and editing, attribute filtering, and saving as reusable named lists.

Capabilities (CP3):

CapabilityAdmin L4Manager L3Staff L2
Import recipient files (.xlsx / .csv) with column mapping✓✓✗
Save and reuse column-mapping configurations✓✓✗
Review / edit / filter the recipient list✓✓✗
Save filtered or edited sets as named custom lists✓✓✗
Edit / clone / delete saved lists✓✓ (own lists only)✗

Import flow — admin uploads .xlsx or .csv; system presents a column-mapping step (file columns left, system fields right — email, first name, last name, matric number, programme, sector, industry, graduation year, etc.; each column mapped or set to "Do not import"); auto-suggestions from header keywords; mapping configurations can be saved for reuse.

Validation — runs after mapping: required field check (email), email format, in-file duplicates, data type on date/numeric/enum fields, match against existing CSM users (linked rows pull merge fields from CSM profile; unmatched rows kept as external). Validation summary: total rows, ready rows, error rows (per-error-type with row numbers), duplicates. Admin can proceed and skip errors, download a corrections file to fix offline and re-upload, or cancel.

Review and edit — list displayed in a table; live recipient count; per-row remove; search by name / email / matric number; manual add via CSM user search or direct email entry; bulk-remove by filter. Edits to saved lists persist; edits in a single-campaign context don't affect any saved source.

Filter attributes — Sector (employer), Industry (employer), Graduation year (student), Programme, School, Campus, Year of study, Activity status (active / inactive / engaged in last 30 days), Booking status, Attendance status (SGCC-related), Custom tags. Filters combine with AND/OR logic.

Saved custom lists — named (with optional description); display name, count, source (imported / filtered / manually curated), last updated date, author. Available as a recipient source in campaign creation (alongside Segments and Ad-hoc — see Campaign Delivery section). Editable, cloneable, deletable (with confirmation; deletion blocked if list is in use by an active scheduled campaign).

Lists vs Segments — Lists (this section) are static: membership fixed at save time, except manual edits. Segments (separate spec) are rule-based: membership re-evaluates against live data. Both are selectable as recipient sources in campaigns.

2.4 Segmentation

For CP3, admins build dynamic student segments using filter rules over student profile, engagement, and institution-defined attributes. Segments are saved as reusable named entities and are selectable as recipient lists for campaigns, filters for analytics, and triggers for bulk actions.

Capabilities (CP3):

CapabilityAdmin L4Manager L3Staff L2
Create / edit / delete custom segments✓✓ (own segments only)✗
Apply institution-defined preset segments✓✓✗
Manage (create / edit / delete) preset segments✓✗ (confirm with Penny — see Open Items)✗
Export segment as CSV✓✓✗

Filterable attributes — three categories:

  • Demographics and profile: Programme, Programme Code, School, School Code, Campus, Year of Study, Expected Graduation Year, Nationality, Gender, Enrolment Status (current / LOA / graduated — confirm with Penny)
  • Engagement and activity status: SGCC Booking Status, SGCC Attendance Status, Resume Submission Status, Internship Status, # Events Attended (last N months), # Jobs Applied
  • Institution-defined labels: At-risk Flag, High Performer Flag (both definitions TBC), Feedback Form Completion Status, Custom Tags (when introduced)

Filter operators — Equals / Not equals (enums, IDs), Contains / Does not contain (text), Greater than / Less than (numeric, dates), In / Not in (multi-value), Is empty / Is not empty (optional fields).

Combining filters — multiple filters combined with AND (all match) or OR (any match); filters can be grouped/nested for complex rules (e.g., (Year = 4) AND (Programme in [FIN, ACC]) AND (Internship Status = no internship)).

Segment save and reuse — each saved segment displays name, optional description, filter rules summary, current member count (live, re-evaluated on open), last updated, author. Admins can edit (changes apply on next use), clone, or delete with confirmation; deletion blocked if the segment is used in an active campaign.

Institution-defined presets — managed centrally; applied with one click. Examples: "All SGCC-eligible students (enrolled in ML0004)", "Students without any SGCC booking", "Final-year students", "Students who missed their last session", "High performers", "At-risk students". Regular admins can apply or clone presets into custom segments, but cannot edit or delete the preset definitions.

Preview and export — on creating/viewing a segment, admin sees the total count and a preview list of the first 25 students; counts update live as filters change. Segment data is exportable as a CSV — see 3.2 Student Segment Export for the full export spec (filename, structure, column order, sample).

Integration — segments are selectable as recipients in email campaigns (see Campaign Delivery), as filters in the analytics dashboard, and as triggers for bulk actions.

2.5 Contact Database Hygiene

For CP3, admins maintain a clean contact database via five connected capabilities: automatic suppression on hard bounce, periodic expunge of chronically invalid addresses, blacklist for specific employers / contacts, tagging recipients into reusable lists from campaign review, and an explicit policy that user accounts are never deactivated by inactivity alone.

Capabilities (CP3):

CapabilityAdmin L4Manager L3Staff L2
View and manage the suppression list✓✓ (view + remove only)✗
Manually remove an address from suppression✓✓✗
Run expunge (on-demand)✓✗✗
Configure expunge criteria + scheduled runs✓✗✗
Add / lift blacklist entries (person or organisation)✓✗✗
Tag selected recipients into existing / new lists from campaign review✓✓✗

Suppression on hard bounce — hard-bouncing emails (permanent delivery failure) are added automatically to a suppression list and excluded from future emails until manually removed. Each entry records: email, suppression reason (e.g., "Hard bounce — mailbox does not exist"), date/time, originating campaign. Soft bounces don't trigger suppression on first occurrence (threshold for repeated soft bounces TBC — see Open Items). Admin can view, search, filter by reason / date, and manually remove (with confirmation).

Periodic expunge — admin permanently removes chronically bounced or invalid addresses. Configurable criteria: address on suppression list more than X days (default 90), hard-bounced N or more times (default 3), matches known invalid patterns (e.g., test@test.com). Runs on-demand (with preview) or on schedule (monthly automated run with email report). Each run requires confirmation and is logged. Expunge is permanent; historical campaign data for expunged addresses is retained (policy confirm — see Open Items).

Blacklist — admin blocks future communications to specific persons (by user ID or email) or entire employer organisations (all contacts under that employer). Each entry has target, required reason, added-by + date, optional end date (e.g., "blacklist for 6 months"). At campaign send time, the system blocks sending if any recipient is blacklisted and shows a warning with the list of blocked recipients; admin can proceed without them. Lift at any time (with confirmation). Record retained for audit and reporting.

Tag contacts into lists from campaign recipient selection — during campaign recipient review (see 2.3 Recipient Management), admin selects one or more recipients and adds them to a list directly. Add to an existing list (search + select) or a new list (create inline). Tagging doesn't modify the current campaign's recipient set — it updates the target list for future reuse. Example: reviewing a campaign to all employers, select 20 priority rows, add to a "Priority employers" list with one click.

Retain inactive users (no deactivation on inactivity alone) — user accounts must not be deactivated or removed solely due to inactivity. Accounts remain active as long as the user is a valid member of the institution (current students per SIS sync, registered employers), regardless of email engagement, login activity, or session bookings. Only the following can change active status: SIS sync marks a student as withdrawn or graduated; admin manually deactivates; blacklist entry (blocks communication, retains record); expunge (removes address after criteria are met). No background job should deactivate users for inactivity reasons. (See Conflicts Flagged for the explicit reversal of Original Requirements 6.1.1.46.1.)

Source: Linear SU-3686 · Original Requirements 6.1.1.30.1, 6.1.1.46.1, 6.1.1.127.1 · Phase 2 Workshop Findings — Communications

2.6 Event Alerts and Digest Preference

For CP3, students choose how they receive event alerts from their Communication Preferences page. The event-matching engine and per-event alert delivery are documented on the Events module page §3 — this section covers the comms-side controls: notification preference, digest cadence, channel routing, and suppression/blacklist integration.

Capabilities (CP3):

CapabilityStudentAdmin L4Manager L3Staff L2
Toggle Event alerts notification preference (on / off)✓———
Choose between per-event alerts OR digest mode✓———
Choose digest cadence (daily / weekly) — admin-configured default applies if no choice✓———
Select preferred channel (Email / WhatsApp / Telegram, when verified)✓———
Configure default digest cadence and send time (institution-wide)—✓✗✗
Configure per-day rate limit cap (over-cap matches roll into digest)—✓✗✗

Notification preference — Event alerts is an optional notification type listed alongside other student-facing Communication Preferences. Students who opt out receive no event alerts; matched events are simply not sent. Mandatory system notifications (e.g., booking confirmations) are unaffected.

Per-event vs digest mode — students choose one of two modes:

  • Per-event alerts — receive an alert each time a matching event opens for registration (subject to the per-day rate-limit cap).
  • Digest — receive a single rolled-up email at a configured cadence. The digest bundles all matching events open for registration since the last cycle, one row per event with details, CTA, and matched interests.

Digest cadence — admin-configured institutional defaults: daily (8am SGT) or weekly (Monday 8am SGT). Students can override the cadence in their preferences. Empty cycles (no matching events) do not send a digest. Mode and cadence changes apply forward only; past alerts are not re-sent.

Channel routing — alerts default to email. Students with verified non-email channels (WhatsApp, Telegram — channel verification scoped in the related Communications Preferences ticket) can route alerts to their chosen channel. Channel routing applies to both per-event alerts and digests.

Suppression and blacklist integration — all alerts (per-event and digest) pass through the suppression list and blacklist before send (see §2.5 Contact Database Hygiene). Suppressed addresses are skipped; blacklisted recipients are blocked with an admin warning at scheduling time.

Rate limit — admin-configurable per-student per-day cap on direct alerts. Over-cap matches automatically roll into the next digest cycle rather than being dropped. The cap default is TBC.

2.7 Job Alert Delivery

Job alerts is an optional notification type in student Communication Preferences; students who opt out receive no job alert emails, with no effect on mandatory or other optional notifications. Pausing or deleting a subscription has the same scoped effect.

Delivery cadence is set per subscription: Immediate (on posting), Daily digest (8am SGT), or Weekly digest (Monday 8am SGT). Empty cycles produce no send; cadence changes apply forward only. Each matching job is delivered at most once across all active subscriptions; immediate alerts are rate-limited to a configurable daily cap (default: 5) with overflow rolling to the next digest. All sends pass through the suppression list and blacklist (§2.5) before dispatch.

Source: Linear SU-3694 · Original Requirements 6.1.1.31.1, 6.1.1.31.3

2.8 AI-Assisted Email Authoring

Admins and Managers can generate AI-assisted body copy and subject line suggestions from within the template editor. The AI is an authoring assistant only — suggestions never auto-apply; admins must explicitly accept, insert, or edit any output before saving or sending.

2.9 Configurable Action Digest

For CP3, admins can configure a digest email that bundles a user’s pending tasks into a single scheduled send. Users receive the digest only when they have pending tasks at send time; the digest respects the communication preferences framework, suppression list, and blacklist throughout.

Capabilities (CP3):

CapabilityAdmin L4Manager L3Staff L2User
Configure digest frequency, send time, template, and task types✓View only (confirm with Penny)✗—

Digest configuration — admin sets: Frequency (Daily at configured SGT time / Weekly on a chosen day and time / Every N days), Send time (SGT), Digest template, Task types to include, and a Minimum task threshold (default: 1; raising this routes low-count scenarios to direct single-task reminders instead). Pending task types (v1): Incomplete profile, Unbooked SGCC bundle, Resume upload pending, Reschedule deadline approaching, Absentee form pending, Feedback form pending, Event RSVP pending. Delivery rules — respects communication preferences (users opted out do not receive the digest unless classified as mandatory), suppression list, blacklist, and an admin option to skip weekends and public holidays.

2.10 Notification Trigger Governance

Admins configure automated notification triggers via a central Triggers view. Only events on an allowed system events list can be used as trigger sources; admins cannot create arbitrary triggers without dev support. Trigger configuration from other tickets (SGCC booking notifications, automation flows, action digest) must use this framework.

Capabilities (CP3):

CapabilityAdmin L4Manager L3Staff L2
Create, edit, and delete triggers✓✗✗
Activate trigger (Draft → Active)✓✗✗
Pause / disable trigger✓✓ (confirm with Penny — see Open Items)✗
View Triggers view and trigger detail✓✓✗

Trigger configuration — each trigger links a named system event to a notification rule (template, audience filter, delay/timing, merge fields, optional conditions). The Triggers view shows: source event, linked template, audience, status (Active / Paused / Disabled / Draft), last fired timestamp, and owner. Allowed system events (v1): Student events (profile completed, SGCC booking confirmed/cancelled/rescheduled, resume uploaded, feedback submitted, mentee signs up); Employer events (account created, job posting published/approved, application received); Event/appointment events (registration submitted/cancelled, appointment booked, attendance marked); System events (SIS sync added/removed user, policy update published).

Trigger states — four non-destructive states: Active, Paused (config preserved), Disabled (config preserved), Draft. Activation from Draft is a distinct permissioned action (Admin L4 only). All state changes and edits are logged with actor, timestamp, and before/after values. Delete requires explicit confirmation. Abnormal firing rates raise an admin alert. Communication preferences, suppression list, and blacklist apply at send time.

2.11 Communication Preferences Framework

Admins classify each system notification type as Mandatory or Optional. Users can toggle Optional types on or off from the Communication Preferences page (accessible from account/profile settings); Mandatory types remain locked and cannot be toggled. All changes are recorded in a read-only audit log. This framework governs send-time enforcement across Campaign Delivery (§2.2), Event Alerts (§2.6), Job Alert Delivery (§2.7), Configurable Action Digest (§2.9), and Notification Trigger Governance (§2.10).

Capabilities (CP3):

CapabilityAdmin L4Manager L3Staff L2 / Student
Classify notification types as Mandatory or Optional✓TBC — confirm with Penny✗
Configure default preference state per role segment✓TBC — confirm with Penny✗
View audit log✓✗✗
View and toggle Optional notification types (Communication Preferences page)——✓

Classification changes take effect immediately across all user sessions. The dispatch engine checks each recipient's preference before send — Optional types are suppressed for opted-out users; Mandatory types send regardless. Admin (L4) can configure default preference states per role segment (Student, Staff, Manager, Admin); defaults apply to new users or users with no saved preference and do not override existing preferences. The audit log is Admin (L4)-only, read-only, no delete.

2.12 Unsubscribe and Suppression Handling

Every optional email includes a visible, accessible unsubscribe link in the footer (clearly labelled, e.g. "Unsubscribe from this list" or "Manage communication preferences"; accessible font size and contrast). The link resolves to the Communication Preferences page (§2.11) where the user completes opt-out in one click. Mandatory communications do not include an unsubscribe link. Branding footer content (institution address, contact info) is included per compliance best practice.

Capabilities (CP3):

CapabilityAdmin L4Manager L3Staff L2 / Student
Unsubscribe link enforcement on all optional emails✓ (system-enforced)——
Choose list/topic-level or global opt-out——✓
Re-subscribe to specific types——✓
Manually resubscribe a contact (on written request only)✓✗✗
View per-campaign and cross-campaign unsubscribe reports✓✓✗
CSV export (cross-campaign report)✓✓✗

Opt-out scope — two levels: List/topic-level (one notification type or category only; other optional communications continue) or Global (all optional communications; mandatory notifications continue). Clicking an unsubscribe link in a specific email defaults to list/topic-level; the preferences page also shows the option to escalate to global. Users can re-subscribe to specific types from the same page.

Automatic suppression — on unsubscribe, the user's preference is updated immediately and their email added to the suppression list with reason "User unsubscribed" and scope recorded. Suppressed recipients are excluded from future sends matching the unsubscribed scope automatically; no admin action needed. User receives an on-screen confirmation explaining what they unsubscribed from, what they still receive, and how to change preferences.

Admin resubscription — admin can manually resubscribe a contact only on written request (confirmation modal + required reason/note + audit log entry with admin name, reason, timestamp). No bulk overrides.

Enforcement — at send time, every recipient is checked against: suppression list (scope-matched), user preferences, and blacklist. Campaign summary shows exclusion counts and reasons. Optional communications cannot bypass opt-out checks; mandatory communications bypass opt-out filtering only (hard bounce / blacklist suppressions still apply to mandatory sends).

Reporting — per-campaign: total unsubscribes, scope breakdown (list-level / global), timeline chart, unsubscribed recipients list. Cross-campaign: aggregate suppressions by reason, unsubscribe rate per notification type/campaign, trend (daily/weekly/monthly), filters by date range/notification type/campaign/segment, CSV export. Full CSV export spec to be added as §3.4 when scoped. Pulls from suppression list (§2.5) and audit log (§2.11).

Source: Linear SU-3707 · Phase 2 Workshop Findings — Communications / Communication Preferences

3.3 Per-Campaign Performance Dashboard

For CP3, every campaign has a per-campaign performance dashboard showing delivery, open, and click metrics, plus per-recipient analytics with status, programme, and role-based tabs. The dashboard supports both one-off campaigns (default view) and recurring campaigns (with a date scrubber to step through individual recurrences).

Capabilities (CP3):

CapabilityAdmin L4Manager L3Staff L2
View per-campaign metric cards (delivery, open, click rates)✓✓✗
View per-recipient analytics with PII (name, email, programme)✓✓ (own workgroup; confirm with Penny — see Open Items)✗
Step through individual sends of a recurring campaign✓✓✗

Table view (in-app) — the dashboard combines metric cards with a per-recipient table. Both render in the in-app analytics view; this section captures the spec only (no CSV export in scope for SU-3688).

Metric cards (no charts) — 8 cards arranged as 2 rows of 4. Row 1: Delivery Rate (%), Unique Opens (count), Total Opens (count), Unique Clicks Received (count). Row 2: Total Clicks (count), Open Rate (%), Click Rate (CTR) (%), Click-to-Open Rate (CTOR) (%). Metric definitions match Communications 3.1 Email Report. Each card has a plain-language hover tooltip. Out of scope (deliberately excluded): unsubscribe rate, bounce rate, spam complaint rate; no time-based engagement charts.

Recipient Analytics table — per-recipient status below the metric cards. Role tabs at the top (Students / Staff / External, each with a count). Stats row: Total Sent + counts per role. Table columns: Name, Email, Last Opened, Programme, Status (badges: Delivered, Sent, Opened, etc.). Keyword search filter on Name / Email. Pagination at bottom.

Recurring campaigns — report header shows recurrence cadence (e.g., "Recurring every Monday at 09:00am") and an end-date indicator. Recurrences sub-list: each send shows Send Date, Delivery Rate, Recipients (count), Status. Date scrubber at the top of the report scrubs through individual recurrences; selecting one updates the metric cards and Recipient Analytics to that send's data.

Source: Linear SU-3688 · Original Requirements 6.1.1.27.1 (Core: "The system shall be able to provide statistics on email open rates, click through rates, student profile/demographic etc. and generate the essential reports.") · Phase 2 Workshop Findings — Communications
