The Forms module enables administrators, managers, and staff to build, publish, and manage forms for collecting structured data from respondents. This spec covers two scopes: SGCC (CP2 — June UAT) and Career Portal (CP3 — September UAT). The Career Portal section documents deltas only — features not present in SGCC scope that must be added for CP3. The SGCC feedback form model (Section 1) also establishes the baseline pattern for how forms behave when attached to other platform modules — including the Events module (see §2.1.14).

Section 1: SGCC Scope (CP2)

The SGCC scope is limited to two feedback forms:

  • SGCC Feedback Form (Resume Critique)
  • SGCC Feedback Form (Mock Interview)

These forms collect post-session feedback from students. All submissions are authenticated. There is one form per session type; all student responses across all SGCC sessions feed into the same form for that type.

1.1 Admin / Manager / Staff

1.1.1 Form Management

Form management covers the full lifecycle of a form from creation through to deletion.

Form lifecycle statuses:

  • Draft — form has been created; not visible to respondents
  • Published — form is live; visible to targeted respondents
  • Withdrawn — form has been taken offline; respondents can no longer view or submit; can be re-published directly by a Manager or Admin
  • Deleted — form has been soft-deleted and moved to the bin; recoverable within 60 days; permanently removed after 60 days

Transitions:

Draft → Published → Withdrawn → (re-published directly, or Deleted)

Draft → Deleted (at any point)

Administrators and Managers can apply the following bulk status updates to multiple forms simultaneously from the forms listing table. Bulk actions are not available to Staff.

ActionApplies ToResulting Status
PublishDraft, WithdrawnPublished
WithdrawPublishedWithdrawn

Notes:

  • Bulk actions are available when one or more forms are selected in the forms table.
  • Only forms in an eligible status for a given action are affected. If a selection contains forms in mixed statuses, the system applies the action only to eligible forms and skips the rest.
  • A confirmation prompt is shown before any bulk action is applied.
  • Staff cannot directly publish or withdraw a form. Those actions are restricted to Managers and Admins.
  • Staff can propose edits to a published form. Edits can be saved as a draft but cannot be published without Manager or Admin action.
  • Forms are scoped to workgroups. Users can only see and manage forms within their own workgroup. The SGCC forms sit within the SGCC workgroup. Admins can see across workgroups.
  • A workgroup is a user-configured team comprising staff, managers, and/or admins. Content created within a workgroup is only visible to members of that workgroup.

Access restriction for SGCC:

SGCC forms are restricted to authenticated students who are enrolled in and have attended an SGCC session. The form does not accept unauthenticated submissions.

Error state — authenticated student not enrolled in SGCC:

If an authenticated student follows the QR code or sharing link but has no SGCC attendance record, the system displays a default error state indicating the form cannot be accessed. The form is not shown.

Suggested default error message: "You don't have access to this form. If you believe this is a mistake, please contact your session coordinator."

1.2 Student

1.2.1 Form Access

  • Students access the SGCC feedback form by scanning a QR code or following a sharing link distributed by the coach during the session.
  • There are exactly two entry points: one QR code / link for Resume Critique and one for Mock Interview.
  • Students must be logged in to access the form. Unauthenticated users who follow the link are redirected to log in before the form loads.
  • If an authenticated student has no SGCC attendance record, they are shown an error message and the form is not displayed (see 1.1.4).

1.2.2 Session Information Display

Upon loading the form, the system looks up the student's SGCC attendance record and auto-populates the following fields. These are displayed as read-only (disabled) so the student can confirm the details before submitting:

FieldSourceEditable by Student
Date of SessionSGCC attendance databaseNo
Time of SessionSGCC attendance databaseNo
Name of CoachSGCC attendance databaseNo

Note: The student's profile information (Full Name, Email, Matric Number, Programme) is captured from their login session and does not appear as visible form fields. It is recorded in the submission automatically.

1.2.3 Feedback Submission

The student-answered portion of the form consists of:

  1. Table scale (required) — 5-point Likert scale (Strongly Agree / Agree / Neutral / Disagree / Strongly Disagree) applied to the following statements:
    • I understood the expectations and outcomes of this session.
    • The objectives were clearly communicated to me.
    • Overall, the session was helpful.
    • I am more informed of the next steps to increase my employability.
    • I am more confident to secure internships/job.
    • I would recommend this session to my peers.
  1. Free-form text (optional) — "Please share any other comments (e.g., what you liked most/least, suggestions for improvement)"
  1. Single-select (required) — "Can we cite your comments when marketing our services?" (Yes / No)

1.2.4 Submission Constraints

  • Each student may submit only one response per form type (one for Resume Critique, one for Mock Interview).
  • The system enforces this constraint. If a student attempts to access a form they have already submitted, they are shown a message indicating they have already submitted feedback for this session type.

1.2.5 Submission Confirmation

  • Upon successful submission, the student sees a confirmation / thank-you message.
  • The submission status is set to Complete immediately upon submission.

2.1.7 Data Integration with Institutional Systems

In SGCC, form responses pull session metadata and student profile fields from the SGCC attendance database and the student login profile (see 1.1.6 and 1.2.2). For Career Portal, this expands into a broader data-integration layer: form responses are synchronised to NTU's centralised data lake (Denodo), made queryable from Qlik Sense for analytics, and can carry contextual fields from sibling modules (Events, Appointments) when a form is linked to a specific Event or Appointment.

Integration capabilities (CP3):

CapabilityAdmin L4Manager L3Staff L2

Form responses queryable in Qlik Sense for analytics dashboards(system)(system)(system)

Configure per-form Qlik connection (field mapping + refresh cadence)✓✗✗

Data flow notes:

  • Form responses sync to NTU Denodo via API at a defined cadence. Cadence value to be confirmed with the data team (see Open Items).
  • Qlik Sense queries the central data warehouse; it does not connect directly to the CSM database.
  • Multi-module joins occur server-side at submission time. The form schema declares the joined fields; admins configure the join in the form's settings.
  • Coach (Staff L2) role does not configure any of this — integrations are admin-defined per form and run automatically once the form is published.
  • Governance (audit logs, edit permissions, status logic) is inherited from existing module specs and not duplicated here. See 1.1.5 / 2.1.6 (audit), 1.1.4 (RBAC), 2.1.4 (status logic).

2.1.9 Sentiment Analysis

For CP3, Forms analytics include AI sentiment classification on free-text responses. Each response is classified Positive / Negative / Neutral and persisted on the response record. The analytics view shows the form-wide sentiment distribution and a sentiment-coloured tag list of the most frequent terms across responses. (See 2.1.3 for the analytics capabilities matrix that gates access.)

Sentiment capabilities (CP3):

CapabilityAdmin L4Manager L3Staff L2
Run sentiment classification on a form on demand✓✗✗
View per-form sentiment distribution and top-term tag list✓✓✓

Behaviour notes:

  • Sentiment classes: Positive, Negative, Neutral. Three-class only; no scoring.
  • Language scope: English only. Non-English responses are flagged Unclassified and excluded from the summary.
  • Re-run: admin-triggered per form. Re-run reclassifies all responses (covers updated responses and provider model changes). No automatic re-runs.
  • Tag list: top-N most frequent terms across responses (stopwords excluded). Each term is coloured by the dominant sentiment of responses containing it.

Behaviour differences by authentication status:

Authenticated respondents:

  • Profile fields (name, email, and role-specific attributes) are auto-filled from their account.
  • Submission is linked to their account record.

Unauthenticated respondents (members of the public):

  • Profile fields are not auto-filled; the respondent enters their details manually where required by the form.
  • At the end of the form, after submission, the respondent is presented with a call to action to create an account or save their details for future use.
  • Submission is recorded without an account link.

2.2.2 Form Access

  • Authenticated respondents access forms via a sharing link or QR code. Unauthenticated users following a link for an authenticated-only form are redirected to log in.
  • For forms configured to allow public (unauthenticated) access, respondents can open and submit the form without logging in.
  • For eligible forms, respondents can save a partially completed response and return later to finish.

2.2.3 Form Guidance and Validation

Not in SGCC scope. Required for CP3.

  • Respondents can see sample answers and hints that illustrate how to respond to a question.
  • Respondents can view links to declarations, terms and conditions, and additional information directly from the form.
  • Respondents receive clear error messages for invalid inputs (e.g. wrong email format, character limit exceeded).
  • Respondents see a countdown of remaining selections when a question has a selection limit.

2.2.4 Multiple Submissions

  • Whether a respondent can submit more than once is controlled by the per-form setting configured by Admins, Managers, or Staff (see 2.1.2).
  • If multiple submissions are disallowed and a respondent attempts to submit again, they are shown a message indicating they have already submitted.

2.2.5 Programme and Application Forms

Not in SGCC scope. Required for CP3.

  • Respondents can fill in structured application forms (e.g. internship applications) that guide them through required fields and document uploads.
  • When logged in, profile details (full name, matric number, programme, school/college, year of study) are auto-filled.
  • Respondents can upload multiple documents separately (e.g. passport, air ticket, declarations) without combining them into a single file.
  • Respondents see a clear confirmation and next steps upon submitting an application.

2.1.13 Group Submission (Team Leader Collect-and-Submit Flow)

For CP3, admins/managers/staff can enable group submission mode on a form. When enabled, one respondent acts as team leader — they initiate the team session, complete team-level sections, generate a share link for their team members to complete member-level sections individually, and submit the consolidated team response to admins once all members have contributed.

This flow is modelled after a group-order pattern: the team leader orchestrates the session; member submissions are staged (not submitted to admins) until the team leader performs the final submit.

Form configuration (admin/manager/staff)

CapabilityAdmin L4Manager L3Staff L2
Enable "Allow team submission" toggle on a form✓✓✓ (requires approval to publish)
Mark each form section as "team-level" or "member-level"✓✓✓ (requires approval to publish)
  • Team-level sections are completed by the team leader and apply to the whole team (e.g. team name, preferred date, session slot).
  • Member-level sections are completed individually by each team member via the share link (e.g. matric number, email, dietary preference).

Team leader — initiating a team submission

  • On opening a group-submission-enabled form, the respondent is prompted to choose: "Submit individually" or "Submit for a team."
  • Selecting "Submit for a team" designates the respondent as team leader and creates a unique team session. The team leader completes all team-level sections first.
  • A unique shareable link is generated for that team session. The link is persistent and accessible to the team leader at any time (e.g. via a "Copy link" button on their submission view). The team leader cannot regenerate the link but can always retrieve it.
  • The share link does not expire unless the form has an expiry date, in which case the link expires when the form does.
  • There is no cap on the number of team members who can submit via the link.

Team members — individual submission via link

  • Access to the share link is gated by the form's visibility settings. If the form is restricted to a specific respondent type (e.g. students only), a member opening the link must log in and be verified as that type before proceeding.
  • Team members complete only member-level sections. Team-level section content is visible to members as read-only context.
  • Each member's submission is saved to the team session but does not trigger a final form submission to admins.
  • Team members can edit their own individual submission at any time before the team leader performs the final submit.
  • Team members can view (read-only) the submissions of other members within the same team session.

Team leader — review and final submission

  • The team leader's view shows a real-time tally of how many members have submitted, with a read-only summary of each member's responses.
  • The team leader can submit the full team response at their discretion — there is no system-enforced minimum member count.
  • Final submission (by team leader only) sends the consolidated team response to admins. Individual member edits are locked after this point.

Admin/manager/staff — responses view for group-submission forms

The responses view for a group-submission-enabled form differs from the standard individual-submission view across all sub-views:

  • Table view: An auto-generated Team ID column appears as the first column, grouping member responses under their respective team. The Team ID is system-generated at the point the team leader initiates the team session.
  • Individual submissions tab: Organised on a per-team basis rather than per individual respondent. Each row represents a team submission, expandable to show each contributing member's member-level section responses. The team leader and each contributing member are identifiable within the expanded row.
  • Summary chart: Includes a pie chart distinguishing between the total number of team submissions and the total number of individual submissions received for the form.

Notifications

  • A confirmation notification is sent to the team leader and all team members upon final submission by the team leader.

2.1.14 Attaching Forms to Events (CP3)

For CP3, a feedback form created in the Forms module can be attached to one or more events. A single form template can be reused across multiple events. This extends the feedback collection model already in place for SGCC (see Section 1 — SGCC Scope) to the Events module.

How it works:

When a form is attached to an event, the system automatically collects respondent metadata and event metadata alongside the form responses. For logged-in respondents (SSO), this data is captured silently — respondents do not need to key anything in. Metadata collected includes:

  • Name
  • Student ID / Matric Number
  • Programme, School, Year of Study
  • Event date
  • Session time

Public events: For public events where participants may not be logged in, metadata fields should be surfaced as optional inputs. Unauthenticated participants may need to self-declare some details. This is a design and dev flag — behaviour to be confirmed before CP3 dev start.

Responses Summary — display rules:

The Responses Summary for event-attached forms follows the same display rules as SGCC feedback forms: only responses to questions authored by the form creator are shown in the summary. Auto-populated respondent and event metadata fields are excluded from the aggregated summary view. This applies consistently across all contexts where a form is attached to a platform module.

Feedback responses are viewable in the Forms module under the relevant form's Responses tab.

2.1.15 Extend Action — Editing a Form's Expiry Date

For CP3, Admin/Manager/Staff can use an Extend action to move a form's expiry date to a later date, for a single form or in bulk. Extend only changes the expiry date — it does not change the form's status. (Bringing an Expired form back to Published is the separate Re-open bulk action — see 1.1.1.)

Decision — which forms are eligible for Extend:

Any form that has an expiry date is eligible, except forms that are already Expired (Expired forms use Re-open instead).

Decision — can the expiry date be cleared (removed) via Extend:

Yes. Leaving the date field empty removes the expiry entirely, so the form stays open with no end date — this mirrors the existing behaviour in form settings.

Risk flagged and accepted: once a published form with responses has its expiry removed, there is no direct way to set a new expiry again — Extend only shows for forms that already have an expiry, and published forms with responses can't be edited via Settings. The only recovery path is to withdraw the form and republish it, which briefly takes it off the student portal. This risk was reviewed and accepted as-is: the withdraw → republish flow is simple, and it's already the established pattern for editing objects that are already published.
