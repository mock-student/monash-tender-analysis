1. Pre-Screening for Event Registration

Event organisers — admins, managers, staff, and employers — can attach a pre-screening form to any event. When enabled, prospective registrants must submit the form before their registration is processed. Pre-screening can be set as required (registration is blocked until the form is submitted) or optional (students may register without submitting).

1.1 Form Configuration

During event setup, the organiser can either select an existing template from the Forms module or create a new inline pre-screening form directly within the event settings page. Inline forms are limited to a single section and support the following field types: text block, single-select, multi-select, free text, and file upload.

1.2 Approval Gate

An approval gate can be enabled alongside pre-screening. When the approval gate is on, a submitted form creates a Registrant record with status Submitted; the student is not yet admitted to the event. When the approval gate is off, a successful submission immediately admits the student as a Participant.

Organiser roles (admin, manager, staff, employer) can update registrant status from Submitted to Approved or Rejected. On approval, the system automatically enrolls the registrant as a Participant and notifies them. On rejection, the organiser may optionally provide a reason; the student is notified of the outcome and, if a reason was provided, it is included in the notification.

1.3 Registrants View

The event management panel includes a Registrants tab (distinct from the Participants tab). The Registrants tab shows all students who have submitted the pre-screening form, along with their status (Submitted / Approved / Rejected). Organisers can open individual records to review responses and approve or reject inline or from the detail view. Bulk approval and bulk rejection are supported; bulk rejection allows an optional shared reason that applies to all selected registrants.

1.4 Student-Facing States

After submitting the form, students see a confirmation that their registration is pending approval (when the approval gate is on). Students can view their submitted responses from their event registration record and receive in-platform (and, if configured, email) notifications when their status changes. If a rejection reason was provided, it is included in the notification.

Scope boundary: The Participants list, including day-of attendance marking, is tracked separately. Inline-created forms are not required to be reusable across events in the initial release.

2. Non-NTU Participant Account Creation via Event Registration

When an event is configured as open to non-NTU participants, the event organiser can enable an "Invite registrants to create an account" toggle in the event settings. When enabled, a post-registration screen is shown to non-NTU registrants with an invitation to create a platform account. Registrants may skip this step; skipping does not affect their event registration record.

If the registrant's email already matches an existing platform account, the post-registration screen instead prompts them to log in rather than creating a duplicate account.

2.1 Account Creation Form

Name and email are pre-filled from the event registration submission and are read-only. The registrant selects an account type from two options, each shown with a short descriptor:

Account TypeDescriptor
EmployerFor organisations looking to connect with NTU talent and post opportunities for students
MentorFor individuals who'd like to give back by volunteering their time to guide NTU students

If Employer is selected, a company name field is shown. As the registrant types, the system performs a fuzzy match against existing employer records and surfaces a confirmation prompt if a close match is found ("We found a matching organisation — is this you: [Company Name]?"). If no match is found, the typed name is accepted as a new company entry. An optional UEN field is shown below the company name; leaving it blank is not a blocker to submission.

The registrant submits the account request without setting a password at this stage.

2.2 Approval and Activation

All non-NTU account submissions are held as pending approval and do not grant immediate access. On submission, the registrant sees a confirmation message and receives a confirmation email acknowledging receipt. On approval, the registrant receives an email with a link to set their password and activate their account.

The admin approval queue for incoming Employer and Mentor account requests is scoped to a separate task cluster (see SU-4430, SU-4431, SU-4432).

3. Interest Tagging, Event Matching, and Alert Delivery for Students

For CP3, students can tag their interests on their profile, the system matches newly published events against those tags, and matching students receive alerts — either per event or rolled up into a digest cadence configured from their Communication Preferences. The feature spans three connected capabilities: profile interest indication, admin-side event tagging, and the matching engine plus alert delivery. The comms-side controls (digest cadence selection, channel routing, suppression/blacklist integration) are documented on the Communications module page §2.6.

3.1 Student Profile Interests

Students indicate interests via an Interests section on their profile with three categories: Industry (e.g. Finance, Tech, Healthcare, Public Sector, Consulting), Function / role (e.g. SWE, Marketing, Data Analytics, HR), Topic areas (e.g. Sustainability, Entrepreneurship, Leadership, AI). Initial vocabulary above is illustrative — confirm with stakeholders before dev start. The tag vocabulary is institution-managed (controlled list maintained by admins in Settings); students select one or more tags per category and can edit at any time. The profile shows currently selected interests with a short helper text explaining usage (e.g., "We use these to send you relevant event alerts. You can change these anytime."). Interests are stored per student and joined to the student's profile record.

Onboarding nudge — on first login after this ships, students see a one-time prompt to set interests. The prompt is dismissible; dismissed students are reminded once more after a configurable delay (default 7 days). Students without interests do not receive event alerts until they opt in from their profile.

3.2 Event Tagging (Admin)

Admins set event tags in the same three categories (industry, function, topic areas) when creating or editing an event. Tags persist on the event record and feed the matching engine. The tag vocabulary is shared with the student profile interests vocabulary — admins maintain a single controlled list in Settings; admins can add, edit, and retire tags.

3.3 Matching Engine and Alert Delivery

Match rule: When an event is published — or an existing event's publish window opens — the system compares each student's interest tags against the event's tags. By default, a student matches if they share at least one tag in any category with the event; the rule is admin-configurable for stricter matching (e.g., AND across categories).

Alert content: matching students receive an alert with event title, date, time, location or online link, short description, matched interests (e.g., "Matches your interest in Finance, Leadership"), and a registration CTA linking into CSM.

Channel routing: alerts default to email but route to verified non-email channels (WhatsApp, Telegram — channel verification scoped in the related Communications Preferences ticket) when the student has selected one in Communication Preferences.

Safeguards:

  • Each student receives at most one alert per event — tag changes after first alert do not auto-trigger a second alert. Admins can trigger a manual one-off re-alert if event tags or audience change significantly.
  • Per-student per-day cap on direct alerts is admin-configurable; over-cap matches roll into the digest (see Communications §2.6).
  • All alerts respect the Event alerts Communication Preference (optional notification type), the suppression list, and the blacklist (see Communications §2.5).

3.4 Admin Analytics

Per event:

  • Match count (students who matched the event)
  • Alerts sent vs. included in digest
  • Open and click rates on alerts — surfaces in the per-campaign performance dashboard (see Communications §3.3)

Per student (visible in the student's activity log):

  • Current interest tags
  • Matched-events history

Admin controls: manage the interest-tag controlled vocabulary, configure the default match rule, configure per-day rate limits and digest send times, trigger a one-off re-alert for a specific event.

3.5 Compliance

Interest data is PDPA-classed personal data. It is used only for matching and alerting; it is not shared with third parties. Clearing all interests immediately stops event alerts; the student remains opted-in to other notifications unless they change Communication Preferences separately (see Communications §2.6).

4. Feedback Forms

Event organisers can attach a feedback form to an event. The form must first be created as a template in the Forms module. Once created, a single form template can be attached to one or multiple events.

When a feedback form is attached to an event, the system automatically collects respondent metadata alongside form responses. For logged-in users (SSO), metadata is collected silently — respondents do not need to key anything in. Metadata collected includes: name, student ID / matric number, programme, school, year of study, event date, and session time.

Public events: For public events where participants may not be logged in, metadata fields should be surfaced as optional inputs. Unauthenticated participants may need to self-declare some details. This is a flag for design and dev to resolve.

Forms Responses Summary — display rules:

The Responses Summary for event-attached feedback forms excludes auto-populated respondent and event metadata fields. Only responses to questions authored by the form creator are displayed in the summary. This is consistent with the behaviour for SGCC feedback forms (see 📋 SGCC §7).
