1. Overview

The SGCC (Small Group Coaching Clinics) module is a standalone module within the CSM platform. It manages the full lifecycle of group career coaching clinics — from session creation and student booking to coach attendance marking and post-session reporting.

The SGCC (Small Group Coaching Clinics) module is a standalone module within the CSM platform. It manages the full lifecycle of group career coaching clinics — from session creation and student booking to coach attendance marking and post-session reporting.

Each SGCC programme cycle consists of two session types that are linked together as a bundle:

  • Session 1 — Resume Critique: Student submits a resume; a coach reviews and critiques it in session.
  • Session 2 — Mock Interview: Coach conducts a mock interview with the student.

By default, students must book both sessions together as a bundle. Unbundled (individual session) booking is only permitted in specific edge cases defined in Section 4.

Roles:

ManagerL3 — ManageApproves absent student reschedule requests, publishes and manages SGCC sessions
StaffL2 — ContributeViews assigned sessions, marks attendance, grades students (5/0), downloads resumes. Staff accounts are used for coaches in SGCC.
StudentL1/L2 — Read/ContributeBooks sessions, uploads resumes, submits absentee forms, reschedules
CSM System—Sends automated emails, enforces business rules, tracks session and booking state
Note: Employer Admin, Employer, and Mentor roles exist in the CSM platform but are not used in SGCC.

2. Session Lifecycle & Statuses

2.1 Session Status Model

Every individual SGCC session moves through the following statuses:

StatusDescription
DraftCreated but not visible to students. Admin can freely edit all fields.
PublishedVisible to eligible students for booking. Sub-statuses: Scheduled, Ongoing, Completed.
WithdrawnTemporarily hidden from students. Students' existing bookings are retained. Admin can make edits, then re-publish. A withdrawn session can also be cancelled.
CancelledSession is soft deleted. Students with bookings are released from the slot and notified. A cancelled session can be restored (via the Restore action in the Sessions table or detail view). Upon restoration, the session will have no active registrations — students must rebook once the session is re-published.

Withdrawal vs Cancellation:

  • Withdrawal is a temporary pause. Students keep their bookings. It exists so admins can make edits (date, time, venue, etc.) without students being able to book during the edit window. Minor changes can be made; students can optionally be notified.
  • Cancellation is equivalent to deleting the session. It releases all student bookings. Students who had signed up are notified and must rebook (if eligible).
  • Students who are booked into a withdrawn session cannot cancel or reschedule until the session is re-published or cancelled.

2.2 Bulk Status Updates

  • Admins can bulk-publish all draft sessions at once.
  • Bulk capacity changes are supported. When bulk-editing sessions, the system must show admins a delta view (what is changing across affected sessions) before confirming.

3. Admin View

3.1 Session Creation

3.1.1 Manual Creation

Admins can create individual sessions manually with the following configurable fields:

  • Session type (Resume Critique or Mock Interview)
  • Date, time, venue/location (online or face-to-face)
  • Coach assignment (a session can have more than one coach assigned)
  • Minimum and maximum student capacity
  • Publish date (controls when the session becomes visible to students)
  • Visibility / student group eligibility (see Section 3.2)
  • Whether the session is listed or unlisted (private)

3.1.2 CSV Bulk Upload

The primary creation method for bulk sessions is CSV upload, reflecting the current workflow of ~1,200 sessions per semester.

CSV template rules:

  • One unified template for both initial creation and updates.
  • Session IDs are included in the downloaded session list export. To update an existing session, include its ID and edit the row. To create a new session, leave the ID blank. Rows without an ID are always treated as new sessions — uploading without IDs on existing sessions will cause duplicates and enrolment data loss.
  • The publish date must be specified in the CSV for each session.
  • Visibility/eligibility conditions can be included directly in the CSV, or configured separately via the session editing UI (both methods are supported).
  • The CSV must include a column to indicate if a session is unlisted (private/not publicly listed to all eligible students).

3.1.3 Ad Hoc Session Creation

Admins can create sessions ad hoc at any time during the semester — for example, to add makeup sessions or accommodate special cases. There is no dedicated "Week 13 makeup" feature; ad hoc creation covers this need.

3.2 Session Editing (Bundled Sessions)

Editable Fields

Admins can edit the following fields on any session:

FieldNotes
DateSubject to the same scheduling constraints as creation — no weekends, public holidays, or NTU university holidays. Must remain within the semester period.
TimeSelected from a dropdown of 3 predefined time slots. Admins can also enter a custom time if none of the preset slots apply.
VenueFor face-to-face sessions — must be a venue from the NTU Maps list. For online sessions — URL can be updated.
CoachMust be an active Staff-role user. Coach availability is checked against existing sessions.
Visibility / eligibility conditionsProgramme codes, year of study, expected graduation year.
Private statusCan be toggled on or off.
Publish dateCan be changed at any time, including backdating to a date that has already passed or changing it after the session is already live. Backdating requires a confirmation modal.
⚠️ Capacity settings are configured globally in the Settings tab and apply to all sessions. They cannot be edited on an individual session basis.

Editing Rules by Session Status

StatusEdit method available
DraftAll fields editable directly with no restrictions.
PublishedAdmins can edit date, time, venue, and coach directly without withdrawing first. Students with existing bookings can optionally be notified by email.
WithdrawnAll fields editable directly. Session is not visible to students during this window.
CancelledNot editable. Must be restored first (see Section 2.1).

Editing Bundled Sessions

An admin can edit date, time, venue, or coach on one session within a bundle without affecting the paired session. Both sessions remain linked as a bundle after the edit. Each session's fields are managed independently — there is no auto-sync.

Two methods are available:

  • Direct edit — admin edits the session while it remains Published. Students are notified of the change (optional).
  • Withdraw then edit — admin withdraws the individual session, makes the edit, then re-publishes. The paired session is not withdrawn and continues as normal.

Conflict Validation on Edit

When saving an edited session, the system runs the same conflict checks as at creation:

  • Coach must not have another session at the same date and time (cancelled sessions excluded).
  • Venue must not have another session at the same date and time (cancelled sessions excluded).
  • Date must not fall on a weekend, Singapore public holiday, or NTU university holiday.
  • Date must fall within the semester period.

If any conflict is found, the edit is rejected and the specific conflict is displayed to the admin.

Student Notification on Edit

When an edit is made to a Published session (both for direct edits and the withdrawal-to-publish flow), the system gives the admin the option to notify enrolled students by email. The notification includes the updated session details and a refreshed .ics calendar file.

RBAC — Who Can Edit Sessions

Admins (L4) and Managers (L3) can make edits to sessions. Staff (L2) cannot edit session fields.

ActionAdmin (L4)Manager (L3)Staff (L2)
Edit session fields (date, time, venue, coach, etc.)✅ Anytime✅ Up to the exact session start time❌ Not permitted
Edit within 3 working days of session start✅ With confirmation modal✅ With confirmation modal❌ Not permitted
Backdate a session✅ With confirmation modal❌ Not permitted❌ Not permitted
Edit once session has started (exact start minute)✅❌ Blocked — clear reason displayed❌ Not permitted

Rules in detail:

  • Editing within 3 working days of session start: Both admins and managers can still make edits during this window, but a confirmation modal is shown before saving to confirm the action is intentional.
  • Backdating: Only admins can set a publish date in the past. This requires a confirmation modal. Managers are blocked from backdating entirely.
  • Once the session has started: The block is triggered at the exact minute of the session's scheduled start time. From that moment, managers cannot make any edits. The system must display a clear reason explaining why editing is unavailable (e.g. "This session has already started and can no longer be edited."). Admins retain edit access.

Session Edit History & Audit Trail

All session edits are tracked and stored as a history log, accessible from the session detail page by Admins, Managers, and Staff.

  • The log records: what was changed, the previous value, the new value, who made the change, and the timestamp.
  • Admins can use this for audit purposes to review the full edit history of any session.
  • The log is read-only and cannot be modified or deleted by any role.

Editing vs. Unbundling

Editing a bundled session (changing its date, time, venue, or coach) is not the same as unlinking. The bundle relationship is preserved after an edit. Unlinking separates the two sessions into independent sessions — see Section 3.3.

3.3 Unbundled Sessions

By default, all SGCC sessions are created and booked as bundles (Resume Critique + Mock Interview). Admins can unlink a bundle to make one or both sessions available individually.

How to Unlink a Bundle

The unlink action is available from the session detail view of either session in the bundle. Unlinking requires the session to be withdrawn first:

  1. Admin withdraws the session to be unlinked. The paired session is not affected and remains Published.
  1. Admin selects the Unlink action from the session detail view.
  1. The two sessions are separated — they are no longer associated as a bundle.
  1. Admin makes any necessary edits to the unlinked session.
  1. Admin re-publishes the session.
⚠️ Unlinking is permanent. Once unlinked, the two sessions cannot be re-bundled.

Admin View After Unlinking

  • Both sessions appear in the sessions list as independent rows with no bundle indicator.
  • Each session can be edited, withdrawn, cancelled, or restored independently.
  • Each session has its own visibility, coach assignment, and publish date.
  • The bundle_link association is removed from the database.

Independence Rules After Unlinking

Once unlinked, each session operates fully independently. There is no auto-sync between them.

ActionBundled behaviourUnbundled behaviour
EditingChanges to one session do not sync to the otherEach session edited independently
CancellationAdmin must cancel the full bundleEach session cancelled independently
CapacityShared across the bundleEach session has its own capacity
VisibilityShared across the bundleEach session has its own visibility rules
⚠️ One remaining dependency: If a student fails the module (marked absent without an approved reason for any session), the remaining session — if not yet attended — is automatically cancelled.

3.3 Admin Dashboard

The admin dashboard provides a high-level operational view including:

  • Enrolled student counts per session
  • Attendance tracking and flagged absences
  • Sessions below minimum capacity (highlighted, with a filter under Sessions view)
  • Bulk actions: assign coaches, publish, withdraw, cancel sessions
  • The sessions list supports column sorting (ascending/descending) and filtering and search by session name, coach name, session type, and status.

Minimum capacity alerts:

  • System highlights sessions where minimum capacity is not met.
  • System notifies admins 5 working days before the session when minimum capacity has not been met.
  • If a session is cancelled/rescheduled due to low attendance, the system sends students a reschedule email.
  • Minimum capacity threshold is configurable in settings.

3.5 Resume Submission Management

If a student does not upload their resume by the deadline:

  • Admin can review the case and either:
    • Allow the student to reschedule (admin always retains this ability)
    • Cancel the student's session slot

Admins can always send a reschedule link to a student regardless of the system's automated state.

3.6 SIS Integration (Student List Sync)

Only students enrolled in NTU module ML0004 are eligible for SGCC. Students who are not enrolled in ML0004 will not be able to see the SGCC module at all.

The system retrieves the ML0004 enrolment list via a daily sync with the SIS (Student Information System). Each day, a refreshed view of the enrolled student list is pulled, and the system reconciles the delta.

⚠️ Enrolment is not finalised at the start of semester. During the first 1–2 weeks of each semester, students can drop the class and new students can enrol. The ML0004 enrolment list will therefore be changing daily during this period. The daily sync must handle this gracefully — additions and removals are expected and normal during this window.

New students (appeared on today's list but not yesterday's):

  • Imported into the SGCC student list and tagged as New.
  • The New tag is visible to admins. The New tag Once an admin has seen the new student entry and navigates away from the page, the New tag is removed.
  • System automatically emails newly added students prompting them to book their SGCC sessions.

Removed students (were on yesterday's list but not today's — i.e. dropped out):

  • Retained in the SGCC student list but marked as Withdrawn.
  • Their existing SGCC session bookings are automatically cancelled.

The system does not permanently delete student records from the SGCC list when they drop out — data is retained for reporting purposes.

3.7 Venue Management

The system integrates with NTU Maps to provide a live, centrally maintained list of all NTU venues, including hyperlinks using the NTU Maps system.

Venue data sourced from NTU Maps includes:

  • Venue name and building
  • Room details
  • NTU Maps hyperlink for each venue

Admin-managed venue capacity:

Within CSM, admins must configure the session capacity for each venue individually through a dedicated venue management interface. This capacity data is not sourced from NTU Maps — it is entered and saved by admins in CSM and stored as part of the venue record. When a session is created and assigned to a venue, the venue's configured capacity is used as a reference.

Staff (coaches) select venues from this list (dropdown) when configuring face-to-face sessions.

3.9 Student Detail View & Activity Log

From the SGCC home page, admins can navigate to the Students tab to view the full student list. The students list supports column sorting and filtering and search — admins can search by student name or matric number, and filter by student attributes including programme and booking status.

Clicking any individual student row opens the Student Detail View.

Student detail view includes:

  • Student profile information: email address, student ID, programme, and other relevant fields
  • SGCC sessions booked by the student
  • Session remarks left by coaches, admins, or managers — each remark can be linked to a specific SGCC session or left unlinked (an SGCC-level remark not tied to a particular session)
  • Overall remarks left on the student's profile (global, not session-specific — see ⚙️ System > Student Remarks)
  • A Student Log tab (see below)

3.8 CSV Re-upload for Published Sessions

When a CSV is re-uploaded while sessions are already published, sessions are updated with the new details while existing student enrolments are retained. Students may be notified of changes depending on what changed.

⚠️ This is separate from the session unlinking flow. See Section 6.2 for unlinking published bundle sessions.

4. Student View

4.1 Session Browsing & Booking (Happy Path)

  1. Student enters the SGCC module and clicks Book Sessions.
  1. Student selects a preferred date range.
  1. The system displays available bundle sessions (Resume Critique + Mock Interview as a pair), filtered to show only sessions assigned to the student's designated coach and eligible based on their programme.
  1. Students only see sessions made available to their programme — not all available sessions.
  1. Student selects a bundle (date/time).
  1. Student is prompted to sync their resume from VMock (see Section 4.3). The student can proceed to book a slot without uploading their resume — this is intentional, as students may be booking early and need more time to prepare their resume.
⚠️ Booking confirmation notice: A student's booking is not confirmed until their resume has been uploaded. The system must display a clear notice of this to the student at the point of booking and in the booking confirmation state. The hard deadline for resume upload is 12pm, 1 working day before the session. Admins reserve the right to cancel the student's session if no resume is submitted by this deadline.
  1. Student confirms booking.
  1. System sends a booking confirmation email with the session details and an Outlook calendar (.ics) attachment.
  1. Session appears in the student's Calendar View.

4.2 Reschedule Rules

  • Students may reschedule up to 1 time per session (configurable in settings).
  • Reschedule is only permitted if made at least 5 working days before the session (configurable).
  • Cancellations made less than 24 hours before the session (configurable) are blocked; the system displays a prompt. Cancellation is not available.
  • If a student with a bundled booking wants to reschedule, they must reschedule the entire bundle — individual sessions within a bundle cannot be rescheduled separately (unless an admin has explicitly unlinked them).
  • Before a student can reschedule or cancel, they must enter a reason. This is required in addition to any deadline or limit restrictions enforced by settings. The reason entered is captured in the student's activity log (see Section 3.9).

4.3 Resume Upload & VMock Integration

Resume deadline rules (confirmed by Joreen):

TriggerAction
3 working days before sessionSystem sends reminder email to student about the upcoming resume deadline
1 working day before session, 12:00pmHard deadline for resume submission
Deadline missedStudent is treated as a no-show (absent). Student must submit the absentee form and go through the absence flow before rebooking.

VMock integration:

  • VMock provides CSM with a list of student resumes that have scored 60 or above. CSM does not have access to resumes below this threshold.
  • VMock sends each student's highest-scoring resume to CSM. CSM makes this available for students and coaches to read in PDF format. Students and coaches will not see the score of this resume.
  • Students can trigger a VMock resume sync from within the CSM system for both Resume Critique and Mock Interview sessions.
  • If no resume exists for a student in the VMock-provided list, the system informs the student that either: (a) they do not have a resume in VMock, or (b) their resume has not yet scored 60 or above.

4.4 Attendance Confirmation

  • The system sends a reminder email 1 day (24 hours) before the session (configurable to X days).
  • Students attend the session.
  • The system sends an attendance acknowledgement email after Session 1, which also includes details of Session 2.
  • After Session 2, the system sends a final attendance acknowledgement email.

4.5 Absence Flow — Session 1 (Resume Critique)

Trigger: Coach marks student as Absent (grade = 0) for Session 1.

  1. System sends a follow-up email to the student requesting justification for absence.
  1. System continues to send daily reminder emails until the student submits the absentee form.
  1. Student fills in the absentee form and attaches supporting documentation.
  1. CAO reviews the submission:
    • Rejected: Student is not allowed to rebook. The student fails the module.
    • Approved: System opens makeup bundle slots for the student and sends an email to re-book a bundle + upload resume.
  1. Student rebooks a bundle (both sessions) and uploads their resume.
  1. Student may reschedule up to 1 additional time (configurable) from this point.
  1. Flow re-enters the Happy Path from Step 6 onwards.
⚠️ Critical rule: If a student misses Session 1 (Resume Critique), their Session 2 (Mock Interview) booking is automatically cancelled. The student must rebook both sessions as a bundle. There is no path to attend Session 2 alone if Session 1 was missed without completing the absentee process.

4.6 Absence Flow — Session 2 (Mock Interview)

Trigger: Coach marks student as Absent (grade = 0) for Session 2.

  1. System sends a follow-up email to the student requesting justification.
  1. System continues to send daily reminder emails until the student submits the absentee form.
  1. Student fills in the absentee form and attaches supporting documentation.
  1. CAO reviews the submission:
    • Rejected: Student cannot rebook Session 2. Student fails the module.
    • Approved: Student can book Mock Interview sessions only (both bundled and unbundled slots) which the student is eligible for.
  1. Student books Session 2 only (unbundled — see Case 1 in Section 5.1).
  1. Student may reschedule up to 1 time from this point (configurable).
  1. Flow re-enters the Happy Path at Session 2.

4.8 Unbundled Session Booking

There are two scenarios where a student books individual sessions rather than a bundle.

Student View After Admin Unlinking

Once an admin has unlinked sessions and made them available individually:

  • Individual sessions appear in the student's session list alongside any bundled sessions, clearly tagged with an "Individual Session" label.
  • Each individual session card states the session type (Resume Critique or Mock Interview).
  • Selecting an individual session — the detail and confirmation UI makes clear this is an unbundled booking.

Independence rules for students on unbundled sessions:

ActionBundled behaviourUnbundled behaviour
BookingBoth sessions booked togetherEach session booked separately, in any order
ReschedulingRescheduling one reschedules the bundleEach session rescheduled independently
CancellationMust cancel the full bundleEach session cancelled independently
Resume syncSingle sync covers both sessionsEach session has its own resume sync
Absentee formAbsence on Session 1 auto-cancels Session 2Absence on one session does not affect the other

If a student has an existing bundled booking and wants to book individual sessions:

The student must first cancel their bundled booking. Once cancelled, they can book individual sessions.

Post-booking confirmation page (first individual session booked):

  • Clearly shows that only one session has been booked.
  • Informs the student which session they still need to book.
  • If they booked Resume Critique → prompt to book Mock Interview, and vice versa.
  • Displays a CTA button to book the remaining session.
📌 When the student clicks the CTA to book their second individual session, only individual sessions of the required type are shown. Bundled sessions are excluded.

Case 1 — Missed Mock Interview After Attending Resume Critique

When it applies: Student attended Session 1 (Resume Critique) AND was absent for Session 2 (Mock Interview), and their absentee request has been approved.

Sessions shown: Mock Interview sessions only (both bundled and individual slots the student is eligible for).

UI behaviour:

  • Session list shows Mock Interview sessions only — no Resume Critique slots shown.
  • Cards render as single-session cards (not bundle cards).
  • Sessions tagged with "Individual Session" label where applicable.

Confirmation page:

  • Standard booking confirmation shown.
  • Resume Critique card remains visible with an "Attended" tag — it does not change.

5. Coach View

5.1 Session View

Coaches can log in to see:

  • All sessions assigned to them
  • Number of students signed up for each session
  • Student resumes submitted at booking (downloadable)

Coach view retains failed/absent students even after rescheduling, to preserve the attendance record.

"New" Student Tag — Session Details Page

A student row in the Session Details page is tagged New if the student was added to the session within 1 working day before the session start time. Working days exclude weekends and Singapore Public Holidays. The tag disappears at the session end time.

The same rule applies to the coach notification email: students added within 1 working day of session start trigger both the New tag and the coach notification email.

Examples:

  • Student added Friday afternoon for a Monday session (Monday is not a public holiday) → New tag shown; Friday is within 1 working day of Monday.
  • Student added Friday afternoon for a Tuesday session (Monday is a public holiday) → New tag shown; Friday is the last working day before Tuesday.
  • Student added Saturday for a Monday session → New tag shown; Saturday falls within the 1-working-day window before Monday.

5.2 Attendance & Grading

Attendance and grade are a single combined field. There is no separate grade component.

ValueMeaning
5Present — student attended
0Absent — student did not attend

Additional fields on the attendance record:

  • Absence reason (dropdown — selectable by coach)
  • Remarks (free text — coach can add notes about the absence or session) (This is a session remark — linked to this specific session registration. For overall student remarks, see ⚙️ System > Student Remarks.)

Coaches mark attendance for Session 1 and Session 2 separately.

Admin access: Admins (L4) can also mark attendance and add remarks on student sessions, for both Session 1 and Session 2.

5.3 Coach Assignment

  • A session can have more than one coach assigned.
  • Coaches can work with students outside their normally assigned schools if an admin configures this.
  • Coach assignment is managed by admins; coaches do not assign themselves to SGCC sessions.

5.5 Unbundled Sessions

When sessions have been unlinked by an admin, coaches see the following:

  • Each session appears as a standalone entry in the coach's session list — there is no bundle indicator.
  • Attendance and grading (5/0) are marked independently per session.

7. Feedback Forms

  • After each session, coaches will ask students to complete a session feedback form.
  • As per CAO processes, two dedicated feedback forms will be created: one for Resume Critique and one for Mock Interview.
  • Feedback forms auto-populate with session data (session name, date, time, coach name) and student data (name, matric number, programme, school, year of study) — students only need to answer the feedback questions themselves.
  • Form analytics include: percentage distribution per option, response count per option, average score, total respondents.
  • Form analytics are exportable to PDF.
  • SGCC feedback forms pull session and student data directly from the SGCC module.

Forms Responses Summary — display rules:

The Responses Summary for SGCC feedback forms excludes auto-populated fields from the summary view. Only responses to questions authored by the form creator are displayed. Specifically:

  • Name of Student — not shown in the summary. Collected at submission but excluded from the aggregated view.
  • Date of Session — not shown in the summary.
  • Time of Session — not shown in the summary.
  • Name of Coach — displayed as a horizontal bar chart showing response count per coach (not as a text list). Designed to support up to ~20 coaches. Figma reference: Coach chart design.

Source: Linear SU-4635

12.3 Key Behaviours Confirmed by Dev

  • Session sub-status logic: Determined by session date relative to current date (SGT, UTC+8 by default). Scheduled = published session with future date. Ongoing = published session occurring today (date-level, not time-level). Completed = published session with past date.
  • Cancelled sessions are soft-deleted: They remain visible in the session list with Cancelled status. Cancellations from CSV re-upload are also shown in the session list (not hidden).
  • Cancelled status is reversible: A cancelled session can be restored to Draft or Published status.
  • Conflict validation: Coach and location availability is checked per date/time slot; cancelled sessions are excluded from conflict checks.
  • Minimum capacity: Stored as a threshold. When registrations fall below minimum, the admin-facing UI shows "Minimum capacity not met." Students are not blocked from booking — the max capacity controls that.
  • CSV notification behaviour on cancellation: To be confirmed. Suggestion is to follow the same notify/don't-notify toggle used in UI-initiated cancellations.
  • Timezone: All dates and times stored and evaluated in SGT (UTC+8).
  • All sessions must fall within the assigned semester's start and end dates. No exceptions.
  • Holiday/weekend validation: Singapore Public Holidays are stored manually (reference: MOM Public Holidays) and NTU academic calendar. Automation of holiday calendar sync is deferred to a future phase.

Role Changes — Staff to Manager

A Manager can also act as a coach and take coaching sessions. When a coach is changed from Staff to Manager, their existing session and student assignments remain valid.

Confirm: Are sessions currently only assignable to Staff users? If yes, this needs to change — Managers must also be valid assignees.

On visibility: No UI change needed for now. Managers (whether or not they are assigned any SGCC sessions) will see all sessions. They can filter by their own name to view their personal sessions. Enhancing this with a dedicated "My Sessions" vs "All Sessions" filter is deferred to CP3 or CP4.
